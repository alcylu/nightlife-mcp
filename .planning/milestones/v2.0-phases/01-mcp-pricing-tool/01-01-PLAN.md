**Step 2: Create `src/services/vipPricing.ts`.**

Export type and function:
```typescript
export type GetVipPricingInput = {
  venue_id: string;
  date?: string;  // "tonight" | YYYY-MM-DD | undefined
};

export async function getVipPricing(supabase: SupabaseClient, input: GetVipPricingInput): Promise<VipPricingResult>
```

Implementation logic (follow the exact flow from RESEARCH Pattern 1):

1. **Validate UUID** — replicate `ensureUuid` locally (copy the UUID_RE + NightlifeError throw pattern from vipTables.ts line 156-162).

2. **Resolve venue** — query `venues` table for `id, name, city_id, vip_booking_enabled, vip_default_min_spend, vip_default_currency`. Throw NightlifeError("VENUE_NOT_FOUND") if not found. Do NOT require vip_booking_enabled (the pricing tool works for all venues).

3. **Resolve city context** — query `cities` table for timezone and cutoff. Default to `Asia/Tokyo` / `06:00` if city_id is null or city not found.

4. **Determine service date** — if `date` is "tonight", call `getCurrentServiceDate(new Date(), timezone, cutoff)`. If YYYY-MM-DD, use directly. If absent, use current service date. Store as `serviceDate`.

5. **Run open-day check** — replicate `resolveClosedDates` logic from vipTables.ts lines 493-590. Pass `[serviceDate]` as the dates array. If `serviceDate` is in the closed set, return early:
   ```typescript
   return {
     venue_id, venue_name, venue_open: false,
     venue_closed_message: `${venue_name || 'This venue'} appears to be closed on ${serviceDate}.`,
     pricing_configured: false, pricing_not_configured_message: null,
     weekday_min_spend: null, weekend_min_spend: null, currency: "JPY",
     zones: [], layout_image_url: null,
     booking_supported: venue.vip_booking_enabled === true,
     booking_note: null, generated_at: new Date().toISOString(),
     service_date: serviceDate, event_pricing_note: null,
   };
   ```

6. **Fetch vip_venue_tables** — query active tables: `supabase.from("vip_venue_tables").select("id,table_code,table_name,metadata,zone,capacity_min,capacity_max").eq("venue_id", venueId).eq("is_active", true)`.

7. **Fetch ALL vip_table_day_defaults** — query ALL rows for venue (no day-of-week filter): `supabase.from("vip_table_day_defaults").select("vip_venue_table_id,day_of_week,min_spend,currency,note").eq("venue_id", venueId)`.

8. **Check per-date overrides** (STATE.md blocker) — when `serviceDate` is set, also check `vip_table_availability` for that specific date: `supabase.from("vip_table_availability").select("min_spend").eq("venue_id", venueId).eq("booking_date", serviceDate)`. If rows exist with non-null min_spend, set `event_pricing_note = "Special event pricing may apply on ${serviceDate} — actual minimums may differ from the general ranges shown."`.

9. **Aggregate pricing** — use the aggregation logic from RESEARCH (Code Examples section). Day-of-week classification: weekend = [5, 6] (Fri, Sat), weekday = [0, 1, 2, 3, 4] (Sun-Thu), matching seeded data patterns.

10. **Determine pricing_configured** — `dayDefaultRows.length > 0 || venue.vip_default_min_spend != null`. If false, set `pricing_not_configured_message = "VIP pricing information is not yet available for this venue."`.

11. **Extract layout_image_url** — replicate `extractLayoutImageUrl` from vipTables.ts line 423-434: parse metadata as object, get `layout_image_url` string, validate it starts with `http`. Use the first table that has one.

12. **Determine booking fields** — `booking_supported = venue.vip_booking_enabled === true`. `booking_note` = null for now (can be enhanced later).

13. **Return VipPricingResult** with all fields populated.

CRITICAL implementation notes (from RESEARCH pitfalls):
- Import from `"zod/v4"` if Zod is needed (it's not needed in the service — only in tool layer)
- Use `.js` extension in all imports: `from "../utils/time.js"`, `from "../errors.js"`, `from "../types.js"`
- `min_spend` from DB can be `number | string | null` — always coerce: `typeof row.min_spend === "string" ? Number(row.min_spend) : row.min_spend`
- `vip_default_min_spend` is also `number | string | null` — same coercion
- Venues with 0 `venue_operating_hours` rows: `hasOperatingHours = operatingHours.size > 0` check means they are NOT blocked
